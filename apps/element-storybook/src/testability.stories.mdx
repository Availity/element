## What makes something easily testable?

HTML, as a construct, is a tree-like document in which a subset of pre-defined tags are used to represent elements on the page. When writing an automated test for such a page, there are a few approaches you can take:

1. Structural
2. Identifiers
3. Contents

### Structural

Structural locators are often the least desirable, since they are brittle and fragile. They are also the most difficult to implement, since they require a deep understanding of the page's structure. This is especially true for pages that are dynamically generated, or that use JavaScript to manipulate the page.

An example is trying to click the `Privacy Policy` link on the login page.

- XPath - `/html/body/div/div/div/div/div/div[1]/div[5]/a/span`
- CSS - `#login-start-page-grid-container > div > div > a > span`

As you can see, these locators are long, and hard to understand why we need them. They give very little context to someone reading the code, and, if poorly documented, can result in a lot of lost time reverse-engineering what is taking place.

### Identifiers

Using identifiers provided by the HTML standard is the preferable approach for making a page more testable. They are not a one-size fits all approach, because they have rules governing where and how they should be used. That said, when available, they should be used, and there are some guidelines to follow.

#### HTML ID Attribute

Referring back to the Availity login page again, we can use an ID selector to discover the Username input field. It is written as `<input id="userId" ...><fieldset ...></fieldset></input>`. This allows us to write the following in Selenium:

- ID - `By.id('userId')`
- CSS - `By.css('input#userId')`
- XPath - `By.xpath('//input[@id="userId"]')`

As you can see, these selectors are much more terse in comparison to an equivalent structural locator. However, the HTML specification has some loose rules about the ID attribute. The rule is that they should be unique within a page. That means, on a dynamic page, if an element can be repeated, or if it shares a name with another element, you might have to come up with a different ID. More than likely, you will be better served using an HTML class.

One of the problems with modern web frameworks is that they can often mangle a well-defined ID, appending a unique identifier for the purposes of tracking it, either as a virtual DOM element, or as something like a build artifact. Since these IDs are not stable, and subject to change on the next page load, or application build, having such an ID will make it functionally useless for the purpose of writing an automated test.

#### HTML Class Attribute

In HTML, a class isn't a formal definition, like you might see in other languages like Java. Instead, it's more like a way of describing a relationship. Elements with the same class should be treated the same. We can see this behavior in CSS libraries, such as Bootstrap and Material UI.

While the Availity login page does not have a good candidate to demonstrate this concept, I will defer to the Claim Status page, which has a concept of patient cards. On the screen, it has a pane of cards that represent claims being processed at the facility. Since each card is equal to the others, defining an ID for them can be difficult, because any criteria might lead to a key collision. While this won't break the page, it will still be a problem for an automated test.

Using a class selector, we can enumerate all patient cards, and then sift through the list of elements in code to find the one we want. That would look like this in Selenium:

- Class Name - `By.className('patient-list-card')`
- CSS - `By.css('div.patient-list-card')`

From there, it is relatively trivial to do something like the following:

```javascript
const patientCardList = await driver.findElements(By.css('div#patientListContainer div.patient-list-card'));

const patientCardDict = new Map();

for (const patientCard of patientCardList) {
  const memberIdElement = await patientCard.findElement(By.css('div.patient-list-content div + div'));
  const memberId = await memberIdElement.getText();

  patientCardDict.set(memberId, patientCard);
}
```

Incidentally, you can see in this snippet from the QA Library how the lack of selectors has led to an ambiguous CSS selector. When selecting the member ID, it was not identified in a meaningful way, so I took the nearest identifiable element, and then looked for some child `div` that had a sibling `div`. If the layout of a patient card ever changes, it might lead to a long debugging session, because it is unclear why we might not have found the correct patient card.

## How should I approach identifiers?

You will be hard-pressed to find a strict guide on how to do what is being described here. Most of it is a matter of opinion, and subject to change based on the needs of the project. That said, there are some general guidelines that can be followed.

1. If something is unique to the page, provide it an explicit ID that is unique to it.
2. If something is repeatable, or you cannot guarantee the uniqueness, try adding an HTML class that describes what the element is for. The name should be descriptive of the element, and only be shared with elements of the same type/structure.
3. If neither of the above are possible, add either a `name` or `data-*` attributes to the element. These attributes are standard HTML attributes, but don't have any specific rules or restrictions on their use. Additionally, they are still usable in a Selenium locator.
